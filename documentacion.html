2. Arquitectura MVC

El proyecto sigue el patrón Modelo–Vista–Controlador:

CapaClasesResponsabilidad
Modelo Equipo, Cellular, EloTelTag, Tablet, ETNube, Territory Almacenan datos y lógica. Las funciones reboteX() y reboteY() permiten a Territory controlar que no salgan del área de simulación.
Vista CellularView, EloTelTagView, TabletView, TerritoryView, FindMyWindow Renderizan cada dispositivo usando QPainter. TerritoryView llama updatePosition() de cada SmartDeviceView.
Controlador MainWindow Reacciona a eventos (Play/Pause/GFindMy) y coordina los pasos de la simulación y vista.

3. Descripción de Clases

Equipo

Clase base para Cellular y EquipoPasivo. Guarda el nombre de su dueño, y el tipo de dispositivo/item. Mediante updatePosition() actualiza su posición. Cuando se llama reboteX o reboteY refleja el angulo actual respecto al eje correspondiente. El movimiento sigue un modelo polar: en cada paso el ángulo varía entre el rango [-deltaAngle, +deltaAngle] y avanza Speed * dTime píxeles. Además almacena en una QList sus posiciones si m_trackingActive esta puesto como true

EquipoPasivo

Clase base para Tablet y EloTelTag. Agrega la capacidad de buscar celulares cercanos mediante findNearbyPhone() y le solicita reportar su ubicación a la nube. Agrega la señal scanAnimation() la cuál es escuchada por smartDeviceView para emitir el efecto de escaneo.

Cellular

Hereda Equipo con nombre fijo "Telefono". Mantiene referencia a ETNube. Reporta su propia posición cada 4 segundos reportLocation() y la posición aproximada (posición propia del celular) de tags/tablets cercanos mediante reportItemPos(Equipo* device).

EloTelTag

Hereda EquipoPasivo. Añade un nombre propio (p.ej. "maleta"). Busca celulares cada 4 segundos.

Tablet

Hereda EquipoPasivo con nombre fijo "Tablet". Busca celulares cada 5 segundos.

ETNube

Almacena dos QMap para guardar por usuario las posiciones de los dispositivos y tags. downloadData(QString usuario) entrega un par de QMap de la información disponible para un usuario en específico. downloadAllData entrega los dos QMap de posiciones disponibles

Territory

Modelo de la simulación. Contiene todos los equipos en un QVector para cada tipo(Cellulars, Tablets, Tags). Si detecta que un equipo está fuera de los margenes de la simulación, llama su función rebote correspondiente.

TerritoryView

Hereda QWidget. Contiene un QPixMap con la imagen de fondo la cual dibuja mediante QPainter, y contiene las vistas de los dispositivos. Propaga el llamado a actualizar las vistas cuando se llama updateView() desde MainWindow.

SmartDeviceView

Hereda QWidget. Es la clase base para cada vista. Modifica el PaintEvent y dibuja según el tipo de dispositivo definido en la clase hija. Si el modelo que recibe se puede castear como EquipoPasivo, conecta la señal scanAnimation() a la función encargada de dibujar el efecto de escaneo.

CellularView / EloTelTagView / TabletView

Vistas de cada dispositivo que heredan de SmartDeviceView. Definen los colores, dimensiones y forma a usar en el PaintEvent. CellularView y Tablet incluyen el menú contextual con "Find My".

FindMyWindow

Hereda de QDialog. Creada como hija de TerritoryView. Ventana flotante que muestra los bienes de una persona. Se actualiza cada segundo cuando recibe la señal updateData() desde TerritoryView solicitando los datos a ETNube. Añade botones para hacer seguimiento a los dispositivos del usuario, estén o no en la nube. Al hacer seguimiento llama setTracking(bool) del equipo correspondiente.

gFindMy

Hereda de QDialog. Creada por MainWindow. Crea un LienzoMapa como hijo.

LienzoMapa

Hereda de QWidget. Muestra una vista del mapa con los dispositivos en su ultima posición reportada en el mapa. Actualiza los datos de posiciones cada un segundo desde ETNube al recibir la señal updateData() de TerritoryView.

3. Dificultades en el proceso de elaboración de la tarea

Se encontró dificultad al implementar ciertas funciones de algunas clases, debido a esto se estimó conveniente crear clases intermedias como EquipoPasivo o SmartDeviceView, lo que evitó tener que repetir código entre varias clases.

Otra dificultad fué que los Tags evitaban la interacción con los tablets y celulares al intentar abrir el menú contextual FindMy, cuando estos se encontraban pasando por encima. Para solucionar esto se hizo que los Tags cambien la propiedad Qt::WA_TransparentForMouseEvents a true. Además se intentó hacer un manejo adecuado de los eventos que no caen dentro del dibujo del dispositivo, para que este pase a otros dispositivos que puedan estar detras.
